## What I learned today (concepts I didn't know before)

**Normal map:** A texture where each pixel's RGB stores which direction that surface is facing. R = facing right, G = facing up, B = facing toward camera. Stored as (128, 128, 255) for a flat surface facing straight at you. With this information, a flat 2D sprite can respond to a moving light as if it has depth.

**N·L dot product:** For each pixel: how directly is the light hitting this surface? If the surface normal and the light direction are aligned (facing each other), the dot product is 1 = full brightness. If they're perpendicular (grazing angle), dot product approaches 0 = dark. That's the whole lighting model in one formula.

**SDF (Signed Distance Field):** For any point in space, it returns how far you are from the nearest shape. Negative = inside the shape. Zero = exactly on the surface. Positive = outside. The shadow shader walks from each pixel toward the light, checking this distance at each step. If it drops below zero, something is blocking the light shadow.

**Sphere tracing (Hart 1996):** Instead of stepping in fixed increments, you step by the SDF value itself if you're 20px away from the shape, you can safely step 20px without missing it. Faster than fixed-step raymarching.

**Soft shadows (Quilez):** Instead of hard shadow/no shadow, accumulate a softness factor during the march: `min(k * h / t, res)`. The closer you come to an occluder, the softer the penumbra. That creeping soft darkness is what makes it feel real.

**Post-processing pass:** After the scene is rendered, run the whole image through another shader. Bloom = bright areas bleed into neighbors (9-tap kernel). Vignette = multiply edges to black via smoothstep. Reinhard tone mapping = prevents whites from clipping. Film grain = hash-based noise added per pixel. Same as Lightroom adjustments but in GLSL.

# Entry 02  The formulas start making sense

## What happened this session

---

## The formulas  what they actually mean

### `length(fragPos - charliePos) - radius`

This is the SDF for a circle. Not magic  pure geometry.

- `fragPos - charliePos`: a vector (arrow) pointing from Charlie Chat's center to the current pixel
- `length(...)`: the distance in pixels between them
- `- radius`: subtract Charlie Chat's body size, because we want distance from the SURFACE, not the center

If I'm 60px from the center and Charlie Chat's radius is 50px, I'm 10px outside the surface. `60 - 50 = 10`. That's the SDF value.

**Positive** = outside. **Zero** = on the edge. **Negative** = inside.

This formula comes from the definition of a circle: all points exactly at distance r from center. That's literally it. No special graphics knowledge. Just distance minus radius.

### `k * h / t`

This runs every step of the shadow ray march, from pixel P toward the light.

- `t` = how far I've walked along the ray so far
- `h` = SDF value at my current position (how far from the nearest obstacle)
- `k` = sharpness constant

The sphere tracing idea (Hart 1996): if the nearest obstacle is 20px away, I can safely step 20px without hitting anything. Big steps in open space, tiny steps near edges. Fast and accurate.

`h/t` approximates the half-angle of the cone from the light that clears the obstacle at this step. Small `h/t` = I nearly grazed something = the pixel is nearly in shadow. Large `h/t` = wide clearance = pixel is lit.

`min(res, k * h / t)` keeps the worst case across all steps. Shadow factor = determined by the closest I came to any obstacle anywhere along the ray.

**The design connection:** low `k` → softer, wider shadows. High `k` → sharper edges. Charlie Chat approaching → lower `k` → shadows spread and soften across the room. That's the horror mechanic. The math does the horror automatically.

---

## Insight: every character casts shadows automatically

Asked whether shadows would work for all moving characters, not just Charlie Chat. Answer: yes, and it's almost free.

The raymarcher evaluates one function  the SDF of the scene. If you define that function as the minimum of all characters' SDFs:

```glsl
float d = min(sdfCircle(p, charliePos, r), sdfCircle(p, npcPos, r));
```

Every character casts shadows. All update automatically. You pass positions as uniforms every frame. No extra loop, no extra passes.

The `min()` works because: if you're near ANY obstacle, you want the distance to be small (indicating proximity). The nearest obstacle always wins.

**Also resolved:** Charlie Chat's SDF must be procedural (not a pre-baked texture) because it moves. A baked texture captures one fixed position. A procedural function recalculates correctly every frame when you pass the new position as a uniform.

# Entry 03  Professor feedback changes everything

## Normal mapping finally clicked

Before the help session, spent time with the GameMaker article and Perplexity. The line that unlocked it:

> "In 2D it will be the normal direction for that particular pixel being shaded."

In 3D, normals come from geometry  the mesh tells you which way each face points. In 2D there is no geometry. Every pixel sits on the same flat plane, so every pixel would have the same normal: (0, 0, 1)  pointing straight at the camera. No depth, no response to light angle, flat cardboard.

The normal map fixes this by storing fake normals per pixel as an RGB image:
- Red = facing right (+X)
- Green = facing up (+Y)
- Blue = facing toward camera (+Z)
- (128, 128, 255) = flat, facing straight at camera

The shader reads this instead of assuming (0, 0, 1) everywhere. Now each pixel has a unique direction and responds to light differently. A cheek curves away from the light. A shoulder catches it. All without any 3D geometry.

**Three ways to make a normal map for a 2D sprite:**
1. Hand-paint RGB colours directly (slowest, most control)
2. Paint a greyscale height map → software generates normals from gradients (fastest, intuitive)
3. 3D model → render normal pass (studio method, overkill)

Method 2 is what we were going to do. The professor has now pushed further.

## The design decision locked today (before professor feedback)

**Environment gets normal mapping. Characters stay flat 2D.**

 The reasoning:

1. **Time.** A 6-frame walk cycle in 4 directions = 24 normal maps to paint by hand. Keeping characters flat cuts art production time by ~80%.

2. **Aesthetics.** Flat cartoony characters in a physically-lit environment creates visual dissonance  the character looks like it doesn't belong in the world. That dissonance is the horror. It reinforces the "uncanny" tone without any extra rendering work.

The professor's feedback confirms this is the right call  environment pipeline is the contribution, not the characters.

## Rendering pipeline locked (4 passes)

```
Pass 1: Environment
  Input: background texture + greyscale height map
  Process: GLSL fragment shader  derive normal from height gradient, compute N·L diffuse
  Output: lit environment to framebuffer

Pass 2: Characters
  Input: flat sprites (Charlie, NPCs)
  Process: draw directly, no shader  maybe distance-based flat tint for basic light response
  Output: composited on top of Pass 1

Pass 3: SDF Shadow (Charlie Chat)
  Input: Charlie Chat's screen position as uniform
  Process: SDF raymarcher  procedural circle SDF, Quilez soft shadow formula
  Output: shadow mask composited over scene

Pass 4: Post-Processing
  Input: full composited scene
  Process: bloom, vignette, film grain, Reinhard tone mapping, chromatic aberration
  Output: final frame
```

This is now the canonical pipeline. Everything gets built to this spec.

## What "procedural normal derivation" means

Instead of providing a pre-computed normal map texture, provide a **greyscale height map** a simple image where white = surface raised toward the camera, black = surface pushed away.

In the GLSL shader, I compute the gradient of this height map using neighbouring pixels (Sobel filter or screen-space derivatives). The gradient tells about the slope. The slope gives the normal direction.

This is more academically interesting because:
- No separate normal map texture to author or store
- The derivation happens live in the shader every frame
- It proves that I understand the relationship between height fields and surface normals  the actual mathematics

I don't fully understand Sobel yet. That's the next thing to learn.

## Step 1: UV gradient

The simplest possible shader. Every pixel outputs its UV coordinate as a color. No logic, no math  just "show me what vTexCoord is."

**What you see:** Black at bottom-left. Red increases to the right. Green increases upward. Blue fixed at 0.5 everywhere. Smooth transitions across the full canvas.

### The code  three files

**`src/index.html`** (one change from original):
```html

<script src="https://cdn.jsdelivr.net/npm/p5@1.9.4/lib/p5.min.js"></script>
<script src="./js/main.js"></script>
```

**`src/js/main.js`:**
```javascript
let myShader;

function preload() {
  // p5.js fetches the shader files over HTTP before setup runs
  myShader = loadShader('shaders/hello.vert', 'shaders/hello.frag');
}

function setup() {
  createCanvas(960, 540, WEBGL); // third arg = WebGL mode, enables GLSL
  noStroke();
}

function draw() {
  shader(myShader);                                        // activate our shader
  myShader.setUniform('uResolution', [width, height]);    // send canvas size to shader
  myShader.setUniform('uMouse', [mouseX / width, mouseY / height]); // send mouse (normalized)
  plane(width, height);                                   // draw fullscreen quad  triggers fragment shader
}
```

**`src/shaders/hello.vert`:**
```glsl
attribute vec3 aPosition;   // vertex position  p5.js provides this automatically
attribute vec2 aTexCoord;   // UV coordinate per vertex  p5.js provides this too
varying vec2 vTexCoord;     // we pass UV to the fragment shader via 'varying'

uniform mat4 uModelViewMatrix;   // camera/world transform  p5.js provides
uniform mat4 uProjectionMatrix;  // perspective/orthographic  p5.js provides

void main() {
  vTexCoord = aTexCoord;
  // Transform vertex position from model space to clip space using p5's matrices
  gl_Position = uProjectionMatrix * uModelViewMatrix * vec4(aPosition, 1.0);
}
```

**`src/shaders/hello.frag` (UV version):**
```glsl
#ifdef GL_ES
precision highp float;
#endif

varying vec2 vTexCoord; // received from vertex shader, interpolated by GPU

void main() {
  // Output UV as color: x = red (left→right), y = green (bottom→top), z = fixed blue
  gl_FragColor = vec4(vTexCoord.x, vTexCoord.y, 0.5, 1.0);
}
```

---

## What I learned: interpolation


The vertex shader runs **4 times** (once per corner of the quad). The fragment shader runs **518,400 times** (960 × 540, once per pixel). The `varying vec2 vTexCoord` value at each corner is:
- Bottom-left: `(0, 0)`
- Bottom-right: `(1, 0)`
- Top-left: `(0, 1)`
- Top-right: `(1, 1)`

The GPU **linearly interpolates** these values across every pixel automatically. A pixel halfway across the screen gets `vTexCoord.x = 0.5`. This is called **barycentric interpolation**  built into the GPU hardware. 

**What decides the gradient size:** the UV range goes 0→1 exactly once across the `plane(width, height)`. To make it steeper, multiply: `vTexCoord.x * 4.0` repeats the gradient 4 times before clamping.

**Why this matters for the project:** the same `varying vec2 vTexCoord` mechanism carries texture coordinates to the fragment shader when we sample the height map for Sobel normal derivation. The principle is identical  just the data changes.

## Step 2: Mouse-driven point light

Updated `hello.frag` to use a `uMouse` uniform instead of UV. Mouse position normalized to 0→1 (same space as UV). Compute Euclidean distance from each pixel to the mouse. Flip it: near = bright, far = dark.

**`src/shaders/hello.frag` (point light version):**
```glsl
#ifdef GL_ES
precision highp float;
#endif

varying vec2 vTexCoord;
uniform vec2 uMouse;        // mouse position, normalized 0→1 by JavaScript

void main() {
  vec2 toLight = uMouse - vTexCoord;   // vector from this pixel to the light (mouse)
  float dist = length(toLight);         // Euclidean distance: sqrt(x² + y²)
  float brightness = 1.0 - dist;       // near = bright (dist≈0), far = dark (dist≈1+)
  gl_FragColor = vec4(vec3(brightness), 1.0); // same value in RGB = greyscale
}
```

**`main.js` change (one line added to draw()):**
```javascript
myShader.setUniform('uMouse', [mouseX / width, mouseY / height]);
```

**What you see:** white circle of light centered on the cursor, falling off to black. Move mouse to a corner  the circle is in the corner. Move to center  it fills the screen more evenly.

## What I learned: uniforms and the data pipeline

- `uniform` = a variable sent from JavaScript to the shader. Same value for every pixel in a frame.
- `varying` = a variable sent from the vertex shader to the fragment shader. Different per pixel (interpolated).
- `mouseX / width` = normalizes mouse to 0→1 to match UV space. Without this, the mouse coords (0→960) would be in a completely different scale than vTexCoord (0→1).

The full data flow this session:
```
JavaScript (CPU)
  → setUniform('uMouse', [...])
    → GLSL uniform vec2 uMouse (GPU)
      → used in fragment shader per pixel
        → gl_FragColor output
          → pixel on screen
```

---

## Why this matters for the project

`length(uMouse - vTexCoord)` is the exact distance calculation used in the point light attenuation in the full pipeline. The only thing missing from the real lighting shader is the **N·L dot product**  which is where the normal map comes in.

Right now: every pixel has the same "fake" normal (pointing straight at camera). The light hits every pixel identically regardless of angle  that's why the falloff is perfectly circular.

When we add the normal map: each pixel will have a unique direction. The same distance calculation stays, but multiplied by `dot(normal, lightDir)`. Surfaces facing the light get bright. Surfaces angled away get dark. That's the whole normal-mapped lighting model.

The pipeline is proven. The next step is the Sobel normal derivation.
